# Relic Media Conventions — Local Discovery Spec

> Companion to `PLAN.md` §4.3 (data model), §5 (Metadata & media), and §8 (budgets).
> Status: **Draft, Phase 1** — defines how the local scanner discovers artwork and video assets without network calls.

---

## 1. Status

**Draft, Phase 1.** This document defines the conventions by which `relic-core` local scanner discovers and indexes local media assets on disk during a library scan. It serves as the specification for the T0 local media discovery pipeline and is consumed by:

- `core/src/scan/` — the scanner module that walks ROM directories.
- `core/src/media/` — the media cache and thumbnailing pipeline.
- `fixtures/` — synthetic and manual test corpuses.

---

## 2. Goals and Non-Goals

### Goals
- **Zero Configuration**: Automatically discover local media files that match standard emulator frontend naming patterns.
- **Offline Purity (T0)**: Run entirely local. No network calls, API keys, or remote scraper dependencies.
- **Robust & Deterministic**: Handle case conflicts, multiple extensions, and path variations predictably.
- **Performant**: Discover files during standard directory traversal without adding substantial disk I/O overhead.
- **Rebuildable**: Keep the DB as the catalog of media availability, mapping to a content-addressed cache that can be fully regenerated at any time from source files.

### Non-Goals
- **Scraping**: Remote network scraping (ScreenScraper, TheGamesDB) is out of scope for the core scanner; it belongs entirely to the `modules/scraper` (T1) module.
- **Editing Media**: Relic does not rename, move, write, or download media files in the user's ROM directory. All scanner operations on the user's library are read-only.
- **Transcoding**: Relic indexes existing file formats; converting video formats (e.g. MKV to WebM) is not handled by the local media engine.

---

## 3. Media Kinds & Definitions

Relic supports four native media kinds. Every discovered file is mapped to one of these types:

| Media Kind | DB representation | Typical Use Case |
|---|---|---|
| **boxart** | `boxart` | The front cover/box art of the game. Main grid asset. |
| **screenshot** | `screenshot` | In-game gameplay capture. Shown on detail screens. |
| **marquee** | `marquee` | Arcade-style cabinet marquee, logo, or overlay header. |
| **video** | `video` | Short gameplay video snap. Played on hover or detail view. |

---

## 4. Lookup Conventions & Priority Order

For any given ROM file located at `<root>/<system-slug>/.../<rom-name>.<rom-ext>` with file stem `<rom-stem>` (the filename without extension), the scanner checks for matching media files. 

When a media kind is sought, Relic searches in the following priority order. The first match found wins:

### 1. Relic-Native Directory (Priority 1)
```
<root>/<system-slug>/.relic-media/<kind>/<rom-stem>.<ext>
```
* **Description**: Relic's own native local storage convention. Placing media here guarantees it takes precedence over all other formats.
* **Layout**:
  - Subdirectories are exactly named after the media kind (`boxart`, `screenshot`, `marquee`, `video`).
  - Example: `C:\games\nes\.relic-media\boxart\Super Mario Bros. (USA).png`

### 2. EmulationStation `gamelist.xml` Relative Paths (Priority 2)
* **Description**: Relic's core `gamelist.xml` importer parses local metadata files generated by tools like Skraper or EmulationStation.
* **Tags mapped**:
  - `<image>` maps to **boxart**.
  - `<marquee>` maps to **marquee**.
  - `<video>` maps to **video**.
* **Paths**: Paths inside the XML elements are resolved relative to the directory containing the `gamelist.xml` file.

### 3. ES-DE (EmulationStation Desktop Edition) Style (Priority 3)
```
<root>/<system-slug>/media/<kind-dir>/<rom-stem>.<ext>
```
* **Description**: The standard layout used by ES-DE.
* **Directory Mapping**:
  - `covers` maps to **boxart**
  - `screenshots` maps to **screenshot**
  - `marquees` maps to **marquee**
  - `videos` maps to **video**
* **Example**: `C:\games\nes\media\covers\Super Mario Bros. (USA).png`

### 4. Same-Folder Convention (Priority 4 - Lowest)
```
<rom-dir>/<rom-stem>.<ext>
```
* **Description**: Simple same-directory naming where the artwork sits right next to the ROM file.
* **Constraints**:
  - Maps exclusively to the **boxart** kind.
  - Restricted to `.png` and `.jpg` / `.jpeg` extensions to prevent accidental matches with non-media files.
  - Example: `C:\games\nes\Super Mario Bros. (USA).png` alongside `Super Mario Bros. (USA).nes`

## 5. Extension Rules & Tie-Breaking

### Permitted Extensions

To avoid parsing or indexing invalid files, the scanner only accepts specific extensions per media class (case-insensitive check):

* **Images (`boxart`, `screenshot`, `marquee`)**: `.png`, `.jpg`, `.jpeg`, `.webp`
* **Videos (`video`)**: `.mp4`, `.webm`

### Case-Sensitivity Rules
- All file stem comparisons are **case-insensitive**.
- A ROM named `Metroid.nes` matches media files named `metroid.png` or `METROID.png`.
- On case-sensitive filesystems (Linux, macOS APFS), the scanner runs a case-insensitive directory search to find matching files.

### Deterministic Tie-Breaking
If multiple files match for the same media kind in the same priority tier (e.g., both `<rom-stem>.png` and `<rom-stem>.webp` exist in the same folder), the scanner selects the best file using these deterministic rules:

1. **Extension Quality Order**:
   - **Images**: `.png` > `.webp` > `.jpg` > `.jpeg`
   - **Videos**: `.mp4` > `.webm`
2. **Lexicographical Order**:
   - If there is still a conflict (e.g. `Metroid.PNG` vs `Metroid.png` on a case-sensitive filesystem), the system breaks ties by sorting the filenames lexicographically via standard byte comparison (preferring lowercase/earlier ASCII values, so `Metroid.png` wins over `Metroid.PNG`).

## 6. Pipeline & Content-Addressed Cache

To ensure fast UI rendering (60 fps grid navigation on low-spec systems or weak GPUs) and prevent lock-ups, discovered media files do not load directly from their raw library paths at runtime. Instead, they flow into a content-addressed thumbnail cache:

```
[Local ROM Library]
   │
   ├─► Discovered Media File (e.g., Zelda.png)
   │     │
   │     ▼ (Scanner calculates content hash)
   │   SHA-256 Hash -> "4f8a9e..."
   │     │
   │     ▼ (Copy/Hardlink/Optimize)
   └───► Cache Store: <cache_dir>/thumbnails/4f/8a9e...webp
```

### Database Representation
The `media` table in SQLite is strictly metadata-driven and references only the hash of the content:
```sql
CREATE TABLE media (
    game_id INTEGER NOT NULL REFERENCES games(id) ON DELETE CASCADE,
    kind TEXT NOT NULL CHECK(kind IN ('boxart', 'screenshot', 'marquee', 'video')),
    source TEXT NOT NULL, -- e.g., 'relic_native', 'gamelist', 'es_de', 'same_folder'
    cache_hash TEXT NOT NULL, -- SHA-256 hash of the media content
    PRIMARY KEY (game_id, kind)
);
```

### Cache Properties
- **Content-Addressed**: Files in the cache are named by their SHA-256 hash. If two games use the identical artwork file, it is stored only once in the cache.
- **Optimization on Import**: High-resolution source images are optionally downscaled and transcoded to optimized formats (e.g., WebP with reasonable dimensions) before placement in the cache.
- **Disposability**: The database contains all parameters needed to locate the source artwork on disk. If the cache directory is deleted, Relic can fully rebuild it by scanning the library, re-calculating hashes, and re-optimizing. No user customization is lost.

## 7. Open Questions

1. **Lazy Hashing for Large Collections**: Hashing every image during initial scan can slow down the `< 30s for 10k files` budget. Should the scanner cache mtime and size in the DB and defer SHA-256 computation until the image is first rendered in the UI?
2. **Video Snaps and Cache**: Should videos be copied into the content-addressed cache, or played directly from their source paths to avoid wasting disk space?
3. **Absolute Paths in gamelist.xml**: How should the scanner handle absolute file paths (e.g. `I:/artwork/nes/mario.png`) in `gamelist.xml` on platforms other than the host that created them?
4. **AVIF Support**: Should AVIF be added to the permitted image extensions list, given its superior compression but higher decoding CPU cost?
